Documentation & Architecture

System Architecture

EdgeHub is an asynchronous telemetry platform designed for distributed infrastructure monitoring. It utilizes a central FastAPI backend to aggregate metrics transmitted by ultra-lightweight, natively compiled Golang agents deployed at the edge.

1. Agent Lifecycle & Authentication

The Edge Agent is a standalone Go binary with zero external dependencies, supporting both AMD64 and ARM64 architectures. It operates with a minimal resource footprint.

Registration Flow:

  • Upon initial execution, the agent utilizes a single-use provisioning token to authenticate against the backend API.
  • Once validated, the backend immediately destroys the token to prevent reuse, registers the hardware metadata, and issues a persistent JSON Web Token (JWT).
  • This JWT is stored locally within the node's filesystem (edgehub-state.json).
  • During subsequent service restarts, the agent bypasses the registration phase entirely, utilizing the stored JWT to immediately resume telemetry transmission.

Security & Revocation:

  • If a node is compromised, administrators can delete it via the EdgeHub dashboard. This instantly invalidates the JWT on the backend.
  • Self-Cleanup Protocol: Upon receiving a 401 Unauthorized response, the revoked agent automatically deletes its local state file and gracefully halts, preventing zombie processes and bandwidth waste.
2. Alerting & State Management

Third-party webhooks (e.g., Discord, Slack) can be configured at the Site level to receive real-time infrastructure alerts. To mitigate false positives caused by transient network instability, EdgeHub implements a buffered state evaluation mechanism for node availability.

  • Threshold Evaluation: When a node fails to transmit a scheduled heartbeat, the backend increments an internal offline_cycles counter. The node enters a failing state, but no external notifications are triggered.
  • Alert Dispatch: If a node accumulates three consecutive missed cycles, the system transitions its state to offline and dispatches the configured Webhook payloads.
  • Automated Recovery: Upon the successful reception of a new heartbeat from a previously offline node, the backend immediately resets the failure counters and dispatches a recovery notification via the established Webhook channels.
3. Deployment Topologies

The agent dynamically adapts its data collection methodology based on the EDGEHUB_MODE environment variable:

  • Linux Native (Systemd): Executed as a root-level background daemon. Designed for bare-metal servers, virtual machines, and IoT devices (e.g., Raspberry Pi).
  • Docker Compose: Operates within an isolated container environment. It requires read-only mounts of host system volumes to accurately calculate underlying hardware utilization.
  • Kubernetes: Can be deployed as a DaemonSet to ensure a single agent instance runs on every physical or virtual node, monitoring the underlying cluster infrastructure.
