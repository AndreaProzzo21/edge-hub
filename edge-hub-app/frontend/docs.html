Documentation & Architecture

System Architecture

EdgeHub is an asynchronous telemetry and Command & Control (C2) platform designed for distributed infrastructure. It utilizes a central FastAPI backend to manage and monitor ultra-lightweight, natively compiled Golang agents deployed at the edge.

1. Agent Lifecycle & Authentication

The Edge Agent is a standalone self-contained binary with a minimal footprint. Its identity is strictly managed by the backend.

  • Provisioning: Upon initial execution, the agent utilizes a single-use token to authenticate. Once validated, the backend immediately destroys the token to prevent replay attacks.
  • Persistence: The agent receives a persistent JSON Web Token (JWT) saved locally in edgehub-state.json. Subsequent restarts bypass registration and resume telemetry seamlessly.
  • Self-Cleanup: If an administrator deletes a node from the dashboard, the token is revoked. Upon receiving a 401 Unauthorized response, the agent gracefully halts.
2. Command & Control (C2)

EdgeHub employs a unidirectional polling architecture, ensuring agents remain secure behind strict NATs or firewalls without requiring open inbound ports.

  • Strictly Scoped Operations: Agents retrieve pending commands seamlessly during scheduled heartbeats. In v1, the only available command is update_jwt.
  • Zero-Downtime Rotation: This single command triggers a secure, atomic rotation of the agent's authentication token using a Rolling JTI strategy, guaranteeing uninterrupted telemetry.
  • No Remote Execution: The C2 implementation is intentionally minimalist. The agent is strictly limited to refreshing its own credentials and possesses no capability to execute arbitrary shell commands, manage system processes, or modify the host OS.
3. Alerting & State Management

Third-party webhooks (e.g., Discord, Slack) can be configured at the Site level to receive real-time infrastructure alerts. EdgeHub implements a buffered evaluation mechanism to prevent false positives.

  • Threshold Evaluation: Missed heartbeats increment an internal offline counter without triggering immediate alerts.
  • Alert Dispatch: Accumulating three consecutive missed cycles forces the node into an offline state, dispatching the configured Webhook payloads.
  • Automated Recovery: A successful heartbeat from a previously offline node immediately resets failure counters and dispatches a recovery notification.
4. Deployment Topologies

The agent dynamically adapts its data collection methodology based on the EDGEHUB_MODE environment variable:

  • Linux Native (Systemd): Executed as a background daemon. Designed for bare-metal servers, VMs, and IoT devices.
  • Docker Compose: Operates within an isolated container. Requires read-only mounts of host system volumes to accurately calculate underlying hardware utilization.
  • Kubernetes: Deployed as a Deployment to interface with the cluster API, while standard Linux native agents can be used to independently monitor individual worker nodes.
